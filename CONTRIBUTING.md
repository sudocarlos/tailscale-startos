# Contributing

## Prerequisites

| Tool | Purpose |
| ---- | ------- |
| Node.js ≥ 20 | TypeScript compilation via `ncc` |
| `start-cli` | Pack `.s9pk` files and inspect manifests |
| `gh` | Create/push GitHub releases (optional) |
| A running StartOS instance | Sideload and test packages |

Install `start-cli`:

```bash
npm install -g @start9labs/start-cli
```

## Build

```bash
# Install dependencies (first time only)
npm install

# Build x86_64 .s9pk
make x86

# Build aarch64 .s9pk
make arm

# Build both
make
```

The build compiles `startos/index.ts` with `ncc` into `javascript/index.js`,
then packs it into a `.s9pk` with `start-cli s9pk pack`.

> **Important:** Never add `console.log` at module level in TypeScript source
> files. The build reads stdout during packing to extract the manifest JSON;
> any extra output will corrupt the pack step.

## Sideload for Testing

1. Build the package: `make x86` (or `make arm` for ARM hardware)
2. Open your StartOS server in a browser (e.g. `http://10.0.0.205`)
3. Navigate to **System → Sideload a Service**
4. Upload the generated `.s9pk` file

## Architecture

### Serve Flow

Serves are managed via the **URL plugin** table actions ("Add Serve" / "Remove Serve") on
service tiles in the StartOS UI.  These actions are not visible in the Actions panel directly.

When a user adds a serve via **Add Serve**:

1. The action reads the current store and assigns a port to the new entry:
   - Auto-assigns starting at 10000, incrementing above the current maximum.
   - The user may supply a **custom port** to override the auto-assigned value.
   - Blocked ports (80, 443, 8080) are rejected.
2. `applyServicesConfig()` (defined in `serves.ts`) is called inside a `SubContainer.withTemp`:
   a. Runs `tailscale serve reset` to clear all existing serves.
   b. For each non-legacy entry, checks the cached `scheme`:
      - `http` / `ws`: runs `tailscale serve --bg --https <port> http://<pkg>.startos:<internalPort>` — Tailscale terminates TLS.
      - `https` / `wss`: runs `tailscale serve --bg --https <port> https+insecure://<pkg>.startos:<internalPort>`.
      - `null` / TCP: runs `tailscale serve --bg --tcp <port> tcp://<pkg>.startos:<internalPort>` — raw TCP forwarding.
      - `packageId = 'startos'`: runs `tailscale serve --bg --https <port> https+insecure://startos.startos:443`.
3. The new port mapping is saved to `startos/store.json`.

When a user removes a serve via **Remove Serve**, the entry is deleted from `store.json`,
then all remaining serves are reset and reapplied in one pass.

On daemon startup, `applyServicesConfig()` is called once (after `BackendState = Running`)
via the `restore-serves` oneshot to restore all non-legacy entries from `store.json`.

**URL export** is handled by `plugin/url.ts`, which reads `store.json` and `status.json`
reactively to build `https://` or TCP URLs for each configured serve.

### Shared Run

The main subcontainer is created with `sharedRun: true`. This causes StartOS to
share `/var/run` between the main container and any temp containers with the
same image ID. This is required so that temp containers used by actions can
reach the tailscaled Unix socket at `/var/run/tailscale/tailscaled.sock`.

### Status Caching

On each successful `tailscaled` health check, the node's Tailscale IP and
MagicDNS name are written to `startos/status.json`. The **URL plugin** (`plugin/url.ts`)
reads this file along with `store.json` to build URLs without shelling out into
a container.

## Making Changes

- **Daemon changes** → `startos/main.ts`
- **Serve apply logic** → `startos/serves.ts`
- **New actions** → add a file under `startos/actions/`, register it in `startos/actions/index.ts`
- **URL plugin / URL export** → `startos/plugin/url.ts`
- **New file models** → add a file under `startos/fileModels/`
- **Manifest metadata** → `startos/manifest/index.ts`
- **Version migrations** → `startos/versions/`

After any change, rebuild with `make x86` and sideload to verify.
